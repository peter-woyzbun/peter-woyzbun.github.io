Datm

Note: I will be filling this section in with more detail shortly. I'm currently working on final documentation for Datm and will be releasing it soon.

Datm is a side project I'm currently working on. It is a graph-based, in-browser platform, written in Python, using the Django web framework, for data manipulation and visualization. The GIFs below demo the main features.

The user creates a project and adds datasets (CSV files), which are represented as nodes on a graph.
The user can apply "transformations" to datasets to create new datasets. A "manipulation set" transformation is an ordered list of manipulations (similar to R's dplyr package), where each manipulation type represents a basic verb of data manipulation (e.g. 'select', 'rename').
The user may also use SQL transformations to create new datasets.
Datasets can be easily joined, and joins are displayed as edges on the project graph.
The user can generate the Python/pandas code required to generate any dataset in their projects. As such, all transformations/manipulations are reproducible outside of Datm.
Any dataset can be easily visualized.

Query Graph

Note: I will be filling this section in with more detail shortly. I'm currently working on final documentation for Query Graph and will be releasing it soon.

Query Graph is a Python package I've been working on as a side project. It's a graph based framework for creating templates for joining data across different types of relational databases (it currently supports PostgreSQL, MySQL, Microsoft SQL, and SQLite). That is, joining data that cannot be joined using a single SQL query.

To illustrate how Query Graph works, I'll use a simple example. Consider the two databases outlined below, each with a single table. The databases use different systems, so one might be MySQL and one might be Postgres. The FLIGHTS table contains arrival delay data for flights departing NYC in 2013. The CARRIERS table maps carrier codes to their actual name.

Suppose we want to query the FLIGHTS table for flights departing to certain destinations, and we also want the carrier names. The YAML below represents a query graph that accomplishes this.


DATABASES:
  flights: [Database credentials]
  carriers: [Database credentials]
QUERY_NODES:
  flights:
      database: flights
      query: |
        SELECT *
        FROM flights
        WHERE dest IN {% destinations|value_list:str %}
      create:
        carrier_lc: str.lowercase(carrier)
  carriers:
    database: carriers
    parent_node: flights
    query: |
      SELECT *
      FROM carriers
      WHERE code IN {{ carrier_lc|value_list:str }}
    join_context:
      how: left
      on_columns:
        carrier_lc: code

The code to execute this query graph in Python, for flights departing to San Francisco ('SFO') or Toronto ('YYZ'), is shown below.


example_query_graph.execute(destinations=['SFO', 'YYZ'])

The diagram below breaks down the query graph execution process into each step.

  1. The FLIGHTS node query is "filled in" with the desired destinations.
  2. The FLIGHTS node query is executed, and a dataframe of results retrieved.
  3. The "carrier_lc" column is created, which is the lowercase carrier codes.
  4. The "carrier_lc" column (unique values) is passed to the CARRIES node.
  5. The CARRIERS node query is "filled in" with the required carrier codes, executed, and a dataframe of results retrieved.
  6. The CARRIERS dataframe is joined with the FLIGHTS dataframe.
